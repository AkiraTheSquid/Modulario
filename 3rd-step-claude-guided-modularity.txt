MODULARIO — Step 3: Claude-Guided Modularity
Feeding structural health back into Claude between tool calls via PostToolUse hook stdout
===========================================================================================

WHAT IT IS
----------
The Layer 1 TUI (step 1) shows YOU the structural health of the codebase.
This step makes CLAUDE see that same information between its own tool calls.

Every time Claude writes or edits a file, the PostToolUse hook already runs the analyzer
and updates state.json (that is what drives the Layer 1 TUI). This step extends that same
hook to also print a compact structural health summary to stdout. Claude Code feeds hook
stdout back into Claude's context as a tool result annotation. Claude therefore sees the
current LOC and DEPS state of the codebase after every single file change it makes —
while it is still mid-task, before it writes its next file.

This is the paper's "Built-in Metrics" and "Structured Support" recommendations combined:
concrete, machine-readable structural data injected into Claude's decision context at the
exact moment it is most actionable.


--------------------------------------------------------------------------------------------
THE KEY MECHANISM
--------------------------------------------------------------------------------------------

Claude Code hooks that write to stdout produce output that Claude sees as part of the
tool result context. This is what makes the feedback loop possible without any changes
to Claude Code itself and without MCP or custom integrations.

The PostToolUse hook already runs modulario-analyze.py and updates state.json.
This step adds one more action to that same hook: print a compact summary to stdout.

Claude sees something like this after each file edit:

  [Modulario] After editing src/components/Editor.js:
    Editor.js       LOC: 312 (+45)   DEPS: 8   STATUS: RED
    ─────────────────────────────────────────────────────
    Session hotspots (red files):
      src/components/Editor.js    LOC: 312   DEPS: 8
      src/store/noteStore.js      LOC: 201   DEPS: 11
    Yellow files: 4   Green files: 17
    ─────────────────────────────────────────────────────
    [!] Editor.js exceeded LOC warning threshold (200). Consider splitting.
    [!] noteStore.js has high DEPS (11). Consider reducing external dependencies.

This output is concrete, not abstract. Claude is not told "avoid tight coupling."
It is told "this specific file has 312 lines and 11 dependencies and is red."
The LMPL paper found that LLMs follow concrete rules reliably and fail on abstract ones.
This is the implementation of that finding.


--------------------------------------------------------------------------------------------
WHAT THE SUMMARY CONTAINS
--------------------------------------------------------------------------------------------

Every PostToolUse hook output includes:

1. The changed file's current stats
   LOC with delta from before the edit (+N or -N)
   DEPS with delta
   STATUS: GREEN / YELLOW / RED

2. All current red files (hotspot list)
   These are the files Claude should be most careful about touching next.

3. Session totals
   Count of red, yellow, green files across the whole target directory.

4. Threshold warnings (only printed if triggered)
   Concrete: "Editor.js exceeded LOC warning (200 lines, currently 312)."
   Concrete: "noteStore.js has high DEPS (threshold 5, currently 11)."
   Not abstract advice — just the numbers and the threshold.

5. Nothing else
   The summary must be short. Claude's context window is finite and every token
   of hook output is context cost. The summary should be under 20 lines.
   No explanations. No suggestions beyond the threshold warnings.
   Claude already has ARCHITECTURE.md and the task prompt — this is data, not advice.


--------------------------------------------------------------------------------------------
FORMAT SPEC FOR HOOK OUTPUT
--------------------------------------------------------------------------------------------

The output format must be:
  - Machine-readable by Claude (structured, consistent)
  - Short (under 20 lines per hook call)
  - Prefixed with [Modulario] so Claude can distinguish it from tool output
  - Consistent across every call (same fields every time, N/A if missing)

Template:

  [Modulario] PostToolUse: {filename}
  ┌─ Changed file ──────────────────────────────────┐
  │ {filename:<40} LOC: {loc:>4} ({loc_delta:+d})   DEPS: {deps:>3} ({deps_delta:+d})   {status} │
  ├─ Red hotspots ──────────────────────────────────┤
  │ {filename:<40} LOC: {loc:>4}   DEPS: {deps:>3}              │
  │ (repeated for each red file, max 5 shown)       │
  ├─ Session totals ────────────────────────────────┤
  │ Red: {n}   Yellow: {n}   Green: {n}             │
  └─────────────────────────────────────────────────┘
  [!] {warning message if threshold exceeded}

If no red files exist: the hotspot section shows "None — all files within thresholds."
If no thresholds exceeded: the warning line is omitted entirely.


--------------------------------------------------------------------------------------------
HOW THIS CONNECTS TO THE OTHER STEPS
--------------------------------------------------------------------------------------------

Step 1 (Layer 1 TUI) and Step 3 share the same PostToolUse hook and the same state.json.
The hook does three things after every file edit:

  1. Run modulario-analyze.py → update state.json         (drives Step 1 TUI)
  2. Print compact summary to stdout                      (drives Step 3 Claude feedback)
  3. (Step 2 handled separately by PreToolUse/Stop hooks)

The hook script (modulario-analyze.sh) is extended in Step 3 to add the stdout print.
No new hook is needed. No new data file is needed. Step 3 is an extension of Step 1,
not a separate system.

Shared data files:
  data/state.json                 written by analyzer, read by TUI and by hook stdout printer
  data/thresholds.json            LOC/DEPS thresholds, used by TUI coloring and by warnings


--------------------------------------------------------------------------------------------
WHAT CLAUDE DOES WITH THIS INFORMATION
--------------------------------------------------------------------------------------------

Claude does not receive instructions from this output. It receives data.
The output does not say "you should refactor Editor.js." It says "Editor.js is red."

What Claude does with that is determined by:
  - The ARCHITECTURE.md rules (Layer 2, not yet implemented)
  - The defensive prompt given at the start of the task (Layer 3, not yet implemented)

For MVP, with only Layer 1 + Step 3 active:
  - Claude sees which files are red after each edit
  - A well-prompted Claude will use that to avoid making red files worse
  - Without Layer 2 and Layer 3, Claude may or may not act on it — but it sees it

As Layers 2 and 3 are added, the defensive prompt will include explicit instructions
like "if Modulario reports a file is RED after your edit, do not add more LOC or DEPS
to that file in subsequent edits without proposing a split first." That instruction
is only meaningful once Claude is already receiving the Modulario data (Step 3).

Step 3 is therefore the prerequisite for Layers 2 and 3, not an optional add-on.


--------------------------------------------------------------------------------------------
WHAT THIS DOES NOT DO
--------------------------------------------------------------------------------------------

- Does not block Claude from making edits (no hard enforcement — advisory only for MVP)
- Does not inject ARCHITECTURE.md rules (that is Layer 2)
- Does not inject defensive prompting (that is Layer 3)
- Does not track tokens (that is Step 2)
- Does not modify Claude Code itself

The only thing Step 3 does is make structural health data visible to Claude in real time.
Everything else — how Claude acts on that data — is controlled by the prompting layers
that come after.


--------------------------------------------------------------------------------------------
BUILD ORDER
--------------------------------------------------------------------------------------------

Build after Step 1 (Layer 1 TUI) is working, because:
  - modulario-analyze.py must exist and be producing accurate state.json
  - You need a real run to verify the hook output format looks right in Claude's context

Step 1: Extend modulario-analyze.sh to print the summary to stdout after updating state.json
  The stdout print reads state.json and formats the summary.
  Test by running the hook manually and inspecting the printed output.

Step 2: Run one real Claude Code session and watch what Claude sees
  Look at Claude's responses — does it reference the Modulario output?
  Does it adjust behavior when a file goes red?
  Note what it ignores — that gap is what Layer 2 and 3 will fill.

Step 3: Tune the output format based on what Claude actually responds to
  If Claude ignores the hotspot list, make it more prominent.
  If the output is too long and eating context, trim it.
  The format should be calibrated against Claude's actual behavior, not just aesthetics.
