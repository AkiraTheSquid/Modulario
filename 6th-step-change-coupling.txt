MODULARIO — Step 6: Change Coupling (Co-Change Pair Detection)
Files that change together repeatedly across sessions, revealing hidden dependencies
===========================================================================================

EMPIRICAL BASIS
---------------
Ease: 30 / 100   Impact: 88 / 100

Change coupling identifies pairs of files that consistently change together in the same
session — which reveals implicit dependencies that the import graph never shows. Two files
with no import relationship can be deeply coupled through shared state, event contracts,
or data format assumptions. Static analysis cannot see this. Only change history can.

The empirical literature is consistent: change coupling correlates with defects at rho=0.46
in multi-project studies and predicts post-release defects better than the entire
Chidamber-Kemerer complexity metrics suite. High co-change pairs explained approximately
45% of post-release defects in the strongest cohort studies.

For AI-assisted development, change coupling is especially diagnostic. When Claude modifies
file A and then file B in the same session repeatedly, it is responding to an implicit
dependency — the codebase is telling you something the import graph isn't. These pairs are
the most likely candidates for either merging (if they represent one logical unit split
across files) or interface extraction (if they should be decoupled but aren't).

The reason ease is 30: this requires multi-session history (Step 5 must be built first),
pair co-occurrence counting across that history, and a threshold to define "frequent
coupling." It is the most data-hungry signal in Modulario. Build it only after Step 5's
history.jsonl has accumulated at least 20–30 sessions of real data.


--------------------------------------------------------------------------------------------
WHAT CHANGE COUPLING TRACKS
--------------------------------------------------------------------------------------------

For every pair of files (A, B) that appeared together in the same session's changed-files
list, track:

  co_change_count:    how many sessions included both A and B as changed files
  sessions_a:         total sessions where A changed (from churn.json)
  sessions_b:         total sessions where B changed (from churn.json)
  coupling_strength:  co_change_count / min(sessions_a, sessions_b)
                      (Jaccard-like: what fraction of A's sessions also changed B?)
  coupling_status:    WEAK / MODERATE / STRONG / CRITICAL

Suggested thresholds (calibrate after real data):
  MODERATE: coupling_strength ≥ 0.3  (changed together in 30%+ of sessions)
  STRONG:   coupling_strength ≥ 0.5
  CRITICAL: coupling_strength ≥ 0.7 AND co_change_count ≥ 5 (avoid noise from rare files)

The coupling_strength formula matters. Raw co_change_count is misleading for frequently-
changed files — a file touched in every session will appear coupled to everything. Dividing
by min(sessions_a, sessions_b) normalizes for frequency. This is the approach used in the
empirical studies that showed the 88/90 impact scores.


--------------------------------------------------------------------------------------------
WHY THIS IS DIFFERENT FROM CIRCULAR IMPORTS
--------------------------------------------------------------------------------------------

Circular imports (Step 4) detect structural coupling that is visible in the code:
A imports B imports A. This is a hard architectural violation.

Change coupling detects behavioral coupling that is invisible in the code:
A and B have no import relationship, but every time you change A you also change B.
This reveals one of three things:

  1. A and B represent a single logical concept split across files —
     they should probably be merged or have a shared interface extracted.

  2. A and B share an implicit contract (a data format, a naming convention,
     a configuration key) that is not encoded in any import or type definition —
     a latent architectural smell that will cause breakage when one changes without the other.

  3. A and B happen to be touched in the same AI sessions because of task scope,
     not because of any real dependency — a false positive that session context can explain.

The human and Claude together need to decide which case applies. Modulario's job is to
surface the pattern; the interpretation is yours.


--------------------------------------------------------------------------------------------
DATA FORMAT
--------------------------------------------------------------------------------------------

New file: state/<hash>_coupling.json

{
  "window_size": 30,
  "last_updated": "2026-03-25T22:00:00",
  "pairs": [
    {
      "file_a": "utilities/current_goals/window.py",
      "file_b": "utilities/current_goals/editor.py",
      "co_change_count": 9,
      "sessions_a": 12,
      "sessions_b": 11,
      "coupling_strength": 0.82,
      "coupling_status": "CRITICAL"
    },
    ...
  ]
}

Only pairs above the MODERATE threshold are stored. Below that, the noise exceeds the signal.
The total number of pairs at MODERATE+ is typically small (under 20) even in large projects
once the threshold is applied — this is not a large data structure.

Built from: state/<hash>_history.jsonl (Step 5). Each line's "changed" list is used to
enumerate all pairs that co-occurred. Pair enumeration is O(k²) where k = files changed
per session. For typical sessions touching 3–15 files, this is trivially fast.


--------------------------------------------------------------------------------------------
HOW CHANGE COUPLING IS SURFACED TO CLAUDE
--------------------------------------------------------------------------------------------

In the PostToolUse [Modulario] block, if the just-changed file is part of a STRONG or
CRITICAL coupling pair, append:

  [~] window.py co-changes with editor.py in 82% of sessions (CRITICAL coupling).
      If you're changing window.py, check whether editor.py also needs updating.
      If they always change together, consider whether they represent one module.

This is advisory, not blocking. Unlike circular imports (which are always wrong), change
coupling may represent intentional co-evolution. The prompt is a check, not a veto.

In the TUI, add a COUPLING panel below the triangle matrix (alongside the VIOLATIONS panel
from Step 4). Show the top 10 strongest pairs with coupling_strength and status. Example:

  ── Change coupling (top pairs) ───────────────────────────────
  window.py         ↔  editor.py            0.82  CRITICAL
  week.js           ↔  week_goals.js        0.71  CRITICAL
  web_main_window.py ↔ main.py              0.54  STRONG
  section_card.py   ↔  widgets.py           0.41  MODERATE


--------------------------------------------------------------------------------------------
RELATIONSHIP TO STEP 5 (CHURN)
--------------------------------------------------------------------------------------------

Step 5 must be built first. Change coupling reads from history.jsonl.

The two signals are complementary:

  High churn, low coupling:  a file changed frequently but alone — unstable interface
                              or a catch-all file that absorbs unrelated changes
  Low churn, high coupling:  a file changed rarely but always with another — tight implicit
                              dependency that rarely surfaces until something breaks
  High churn, high coupling: the worst case — file changes constantly and drags others with it

The combination of churn status and coupling status gives a richer picture than either alone.
Consider adding a combined "coupling-adjusted churn risk" score to the TUI summary in a
later iteration.


--------------------------------------------------------------------------------------------
WHAT THIS DOES NOT DO
--------------------------------------------------------------------------------------------

- Does not explain WHY two files co-change — only that they do. The human interprets this.
- Does not detect coupling at the function or method level. File-level is the resolution.
- Does not replace static dependency analysis (Steps 1-4). Files that import each other
  AND co-change are unsurprising. The interesting signal is files that co-change WITHOUT
  an import relationship — those are the hidden dependencies.
- Does not work well with fewer than 20 sessions of history. Below that threshold, the
  co-occurrence counts are too noisy to trust. Do not build this until Step 5 has
  accumulated real data.

CHANGES TO EXISTING FILES
--------------------------------------------------------------------------------------------

modulario-analyze.py
  - After updating churn.json (Step 5), enumerate all file pairs from history.jsonl
    within the window
  - Compute co_change_count and coupling_strength for each pair
  - Filter to pairs above MODERATE threshold
  - Write coupling.json
  - In build_claude_summary(), check if changed file is in any STRONG/CRITICAL pair
    and append [~] coupling warning

modulario-tui.py
  - Load coupling.json alongside state.json and churn.json
  - Add COUPLING panel below VIOLATIONS panel
  - Show top 10 pairs ranked by coupling_strength
  - Pairs involving a currently-selected file are highlighted

thresholds.json
  - Add "coupling_window": 30
  - Add "coupling_thresholds": {"moderate": 0.3, "strong": 0.5, "critical": 0.7}
  - Add "coupling_min_count": 5  (ignore pairs with fewer co-changes to reduce noise)


--------------------------------------------------------------------------------------------
BUILD ORDER
--------------------------------------------------------------------------------------------

Prerequisite: Step 5 must be complete and history.jsonl must have ≥ 20 real sessions.
Do not attempt to build Step 6 before that data exists — the signal will be noise.

Step 1: Implement pair enumeration from history.jsonl
  For each session line in history.jsonl (within window), generate all file pairs
  from the "changed" list. Accumulate co_change_count per pair.
  Test: manually create a history.jsonl with known co-change patterns and verify counts.

Step 2: Compute coupling_strength and filter
  coupling_strength = co_change_count / min(sessions_a, sessions_b)
  Filter to pairs above the MODERATE threshold.
  Write coupling.json.
  Test: inspect coupling.json for a real project after 20+ sessions. Do the top pairs
  make intuitive sense? Adjust thresholds.json if results seem noisy.

Step 3: Add coupling warnings to PostToolUse output
  In build_claude_summary(), look up the changed file in coupling.json.
  If it appears in a STRONG or CRITICAL pair, append the [~] advisory line.
  Test: trigger the hook on a file known to be in a coupling pair.

Step 4: Add COUPLING panel to the TUI
  Load coupling.json. Render the top-10 panel below VIOLATIONS.
  Highlight pairs where either file is the currently-selected TUI row.

Step 5: Interpret first results and tune thresholds
  After the first real coupling.json with 20+ sessions of history:
  - Are the CRITICAL pairs surprising or expected?
  - Are there pairs with no import relationship? Those are the most interesting.
  - Do the thresholds need adjusting?
  Document findings in a short note alongside the data files.
