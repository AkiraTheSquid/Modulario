MODULARIO SESSION TUI — Plan
Session token usage and response tracking, synced to Claude Code via Stop hook
===========================================================================================

WHAT IT IS
----------
A second terminal window TUI that tracks the current Claude Code session in real time.
Every time Claude finishes a response, the Stop hook fires, writes data to a log file,
and this TUI re-renders to show the updated session history.

This TUI has nothing to do with the filesystem or directory health.
That is handled by the separate Layer 1 TUI (simple-modulario-plan.txt).

This TUI answers one question: what is this Claude session costing, and is it improving
the codebase or making it worse?


--------------------------------------------------------------------------------------------
WHAT IT DISPLAYS
--------------------------------------------------------------------------------------------

TWO PANELS

Panel 1 — Session Summary (top, fixed height)
----------------------------------------------
Shows the running totals for the current session:

  Session: Delta-Note-Local  |  Started: 14:21  |  Responses: 12
  ─────────────────────────────────────────────────────────────────
  Total tokens in:   38,421     Avg per response:   3,201
  Total tokens out:   7,883     Avg per response:     657
  Total combined:    46,304

  Health: 2 red → 1 red  (improved)   Files changed this session: 6

Panel 2 — Response Log (bottom, scrollable)
--------------------------------------------
One row per Claude response, most recent at top:

  #    Time    Tokens In   Tokens Out   Files Changed        Health Delta
  ─────────────────────────────────────────────────────────────────────────
  12   14:32   4,821        892         Editor.js             2 red → 1 red
  11   14:28   2,103        441         noteStore.js          +1 yellow
  10   14:21   1,876        318         (none)                —
   9   14:18   5,102       1,204        Sidebar.js, utils     1 red → 2 red
   8   14:11   3,241        728         (none)                —

Color coding for Health Delta column:
  green  = net improvement (fewer red/yellow files after than before)
  red    = net regression (more red/yellow files after)
  dim    = no file changes or no health change

The Health Delta is computed by comparing the Layer 1 TUI's state.json snapshot
taken before the response (at PreToolUse start) vs. after (at Stop).
This links the two TUIs together through shared data files without either
TUI depending directly on the other.


--------------------------------------------------------------------------------------------
HOW IT SYNCS WITH CLAUDE CODE
--------------------------------------------------------------------------------------------

TWO HOOKS ARE NEEDED

Hook 1: PreToolUse (fires before Claude starts using tools in a turn)
  - Snapshots the current state.json from the Layer 1 analyzer
  - Saves it as data/session-snapshot-before.json
  - This captures the structural health state BEFORE Claude makes changes

Hook 2: Stop (fires when Claude finishes a complete response turn)
  - Receives JSON payload via stdin with session metadata
  - Extracts: input tokens, output tokens, session ID
  - Reads data/session-snapshot-before.json (health before)
  - Reads current data/state.json (health after, already updated by PostToolUse hook)
  - Computes health delta (red file count before vs. after)
  - Appends one row to data/session-log.json
  - TUI re-renders

DATA FILE: data/session-log.json
  Array of response entries, each:
  {
    "response_num": 12,
    "time": "14:32:07",
    "tokens_in": 4821,
    "tokens_out": 892,
    "files_changed": ["src/components/Editor.js"],
    "red_before": 2,
    "yellow_before": 4,
    "red_after": 1,
    "yellow_after": 4,
    "health_delta": "improved"
  }

DATA FILE: data/session-meta.json
  Running totals for the session summary panel:
  {
    "session_name": "Delta-Note-Local",
    "started": "14:21:03",
    "response_count": 12,
    "total_tokens_in": 38421,
    "total_tokens_out": 7883
  }

Both files live in /home/stellar-thread/Applications/Modulario/data/
and are reset at the start of each new Claude Code session.


--------------------------------------------------------------------------------------------
HOOK CONFIGURATION
--------------------------------------------------------------------------------------------

In the target project's .claude/settings.json (or ~/.claude/settings.json globally):

  {
    "hooks": {
      "PreToolUse": [
        {
          "matcher": ".*",
          "hooks": [
            {
              "type": "command",
              "command": "/home/stellar-thread/Applications/Modulario/scripts/session-snapshot.sh"
            }
          ]
        }
      ],
      "Stop": [
        {
          "hooks": [
            {
              "type": "command",
              "command": "/home/stellar-thread/Applications/Modulario/scripts/session-stop.sh"
            }
          ]
        }
      ]
    }
  }

session-snapshot.sh:
  Copies data/state.json to data/session-snapshot-before.json
  Idempotent — safe to call on every tool use, only the first one per turn matters.
  Uses a turn-ID sentinel file to skip subsequent calls within the same turn.

session-stop.sh:
  Reads stdin JSON payload
  Extracts token counts
  Reads session-snapshot-before.json and current state.json
  Computes health delta
  Appends to session-log.json and updates session-meta.json
  Clears the turn-ID sentinel so the next PreToolUse captures a fresh snapshot

NOTE ON TOKEN PAYLOAD
---------------------
The Stop hook receives a JSON payload via stdin. The exact fields need to be verified
when building session-stop.sh. If token counts are not directly in the payload,
the fallback is to parse them from Claude Code's output or use available usage fields.
This is the one unknown to resolve at build time — everything else is deterministic.


--------------------------------------------------------------------------------------------
THE TUI ITSELF
--------------------------------------------------------------------------------------------

Script: /home/stellar-thread/Applications/Modulario/tui/session-tui.py

Run with:
  python3 session-tui.py --session Delta-Note-Local

Watches data/session-log.json and data/session-meta.json via inotify (watchdog library).
Re-renders both panels on every file change.

Built with Python rich (same dependency as the Layer 1 TUI).
  rich.Layout  — two-panel split
  rich.Table   — scrollable response log
  rich.Live    — auto-refresh loop

The two TUIs share the same Python dependencies and the same data/ directory.
They do not talk to each other directly — they communicate only through files.


--------------------------------------------------------------------------------------------
DIRECTORY ADDITIONS (on top of simple-modulario-plan.txt)
--------------------------------------------------------------------------------------------

scripts/
  session-snapshot.sh           PreToolUse hook: snapshots state.json before each turn
  session-stop.sh               Stop hook: writes response entry to session-log.json

tui/
  session-tui.py                the session TUI, run in a second terminal

data/
  session-log.json              one entry per Claude response (appended by Stop hook)
  session-meta.json             running session totals
  session-snapshot-before.json  health snapshot taken at start of each turn
  turn-sentinel                 empty file whose presence means snapshot already taken


--------------------------------------------------------------------------------------------
RELATIONSHIP BETWEEN THE TWO TUIs
--------------------------------------------------------------------------------------------

Layer 1 TUI (simple-modulario-plan.txt):
  - Watches the filesystem (the target codebase)
  - Triggered by PostToolUse hook after file writes/edits
  - Shows structural health: LOC + DEPS per file, color-coded
  - Completely independent of the Claude session
  - Would work even without Claude running at all

Session TUI (this file):
  - Watches the Claude session via Stop hook
  - Triggered by Stop hook after each complete response
  - Shows token cost and health trajectory per response
  - Depends on data/state.json from the Layer 1 analyzer for health delta

They share:
  data/state.json         written by Layer 1 analyzer, read by session-stop.sh
  Python dependencies     rich, watchdog

They do not share:
  Hook triggers           different events
  TUI code                separate scripts
  Display concerns        separate windows


--------------------------------------------------------------------------------------------
BUILD ORDER FOR THIS TUI
--------------------------------------------------------------------------------------------

Build AFTER the Layer 1 TUI is working, because:
  session-stop.sh depends on data/state.json existing and being accurate
  You need real data to verify the health delta computation is correct

Step 1: Write session-stop.sh
  Test standalone by simulating a Stop hook call with fake JSON piped to stdin.
  Verify it writes correct entries to session-log.json.

Step 2: Write session-snapshot.sh
  Simple file copy plus sentinel logic. Test that it is idempotent.

Step 3: Wire up both hooks in settings.json

Step 4: Write session-tui.py
  Test with a manually populated session-log.json before running a real session.

Step 5: Run a real Claude Code session with both TUI windows open side by side.
  Verify the session TUI updates after each response with correct token counts
  and correct health delta pulled from the Layer 1 state file.
